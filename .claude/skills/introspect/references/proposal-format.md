# Harness Evolution Proposal Format

Proposals are generated by the monthly-evolve workflow and written to
`~/qara/thoughts/shared/introspection/reports/proposals-YYYY-MM.md`.

JM reviews and applies proposals manually. Proposals are NEVER auto-applied.

## Structure

Each proposal file contains one or more proposals in this format:

```markdown
### Proposal #N: [Title]

**Type:** hook | skill | setting | workflow | miner | new-component
**Target file(s):** `path/to/file`
**Action:** Add | Modify | Remove | Create
**Evidence:** [pattern name] at [confidence] tier, [N] observations
**Risk:** low | medium | high
**Reversible:** yes | no

**Problem:**
[1-2 sentences: what the pattern data shows]

**Proposed Change:**
[Exact code diff or configuration change, fenced in code blocks]

**Expected Impact:**
[What should improve, how to verify]

**Rollback:**
[How to undo if the change causes problems]
```

## Field Definitions

| Field | Required | Description |
|---|---|---|
| Type | Yes | Category of change — determines review rigor |
| Target file(s) | Yes | Exact paths of files to modify |
| Action | Yes | What kind of modification |
| Evidence | Yes | Which pattern triggered this, at what confidence tier |
| Risk | Yes | low = config/doc change; medium = behavior change; high = architecture change |
| Reversible | Yes | Can this be `git revert`ed cleanly? |
| Problem | Yes | What the data shows — observations, not opinions |
| Proposed Change | Yes | Exact diff or new content — not vague suggestions |
| Expected Impact | Yes | Measurable prediction (e.g. "error rate should drop below 1%") |
| Rollback | Yes | Specific steps to undo (file path, command, or git revert) |

## Rules

- One proposal per discrete change — don't bundle unrelated fixes
- Evidence must reference actual pattern files with confidence tiers
- Proposed Change must be exact (copy-pasteable), not "something like this"
- Risk rating drives review depth: low = skim, medium = read carefully, high = test first
- Proposals with risk=high should include a test plan
- Each proposal gets a sequential number (#1, #2, ...) within the monthly file
- When JM approves a proposal, add an entry to `IMPROVEMENT_LOG.md`

## Proposal Types

| Type | Examples | Typical Risk |
|---|---|---|
| hook | Enrich logging, add new hook, fix timeout | medium |
| skill | Update workflow, add routing, new skill | medium |
| setting | Timeout change, permission adjustment | low |
| workflow | Update prompt instructions, add step | low |
| miner | Add extraction function, new report field | medium |
| new-component | New CLI tool, new lib, new pattern file | high |

## Integration

- Monthly-evolve workflow generates proposals based on confirmed/established patterns
- Proposals are written to `reports/proposals-YYYY-MM.md`
- Monthly report includes a summary table linking to each proposal
- Approved proposals get logged in `IMPROVEMENT_LOG.md`
- Significant proposals also get a `DECISIONS.md` entry
- Both follow `.claude/CLAUDE.md` § Documentation Hygiene (Agent-Facing): concise prose, 1–3 sentences per section, archive-by-year at `<stem>/YYYY.md` when ≥400 lines
